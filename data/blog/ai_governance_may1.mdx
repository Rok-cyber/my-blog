AI는 이제 답변을 생성하는 도구를 넘어 검색하고, 파일을 읽고, 코드를 실행하고, 외부 시스템을 호출하는 **행위자**에 가까워지고 있습니다. 중요한 질문도 “얼마나 똑똑한가”에서 “어떤 권한을 주고, 결과를 어떻게 검증할 것인가”로 이동합니다.

---

## 1. 모델 이름보다 중요한 변화: 도구를 쓰는 AI

OpenAI의 공식 문서는 모델 응답에 웹 검색, 파일 검색, 함수 호출, 원격 MCP 같은 도구를 연결할 수 있다고 설명합니다. 모델은 주어진 도구 가운데 필요한 것을 선택하고, 애플리케이션은 `tool_choice` 같은 설정으로 그 범위를 통제할 수 있습니다.

이 변화의 핵심은 벤치마크 점수보다 **모델이 실제 시스템에 미치는 영향의 범위**입니다. 읽기만 가능한 에이전트와 파일을 수정하거나 외부 서비스를 호출할 수 있는 에이전트는 전혀 다른 위험 프로필을 가집니다.

<InfoBox title="Governance Insight">
  에이전트의 위험은 모델 성능 하나로 결정되지 않습니다. **접근 가능한 데이터, 사용할 수 있는 도구,
  실행 전 승인 절차, 실행 후 로그**를 함께 설계해야 합니다.
</InfoBox>

## 2. 권한을 네 단계로 나누기

조직이 에이전트를 도입할 때는 “사용/미사용”의 이분법보다 권한을 단계별로 나누는 편이 현실적입니다.

| 단계         | 가능한 행동                | 필요한 통제                       |
| :----------- | :------------------------- | :-------------------------------- |
| 1. 제안      | 초안, 요약, 선택지 제시    | 출처 표시와 사람의 검토           |
| 2. 조회      | 문서·DB·웹 검색            | 최소 권한, 민감정보 필터          |
| 3. 변경      | 코드·파일·업무 데이터 수정 | 변경 범위 제한, diff와 승인       |
| 4. 외부 실행 | 배포·결제·메시지 전송      | 명시적 승인, 취소 수단, 감사 로그 |

에이전트가 더 많은 일을 할수록 사람의 역할이 사라지는 것은 아닙니다. 사람의 역할은 모든 단계를 직접 수행하는 데서, **경계와 승인 조건을 설계하고 예외를 판단하는 일**로 이동합니다.

## 3. 감사 가능성은 기능이다

에이전트 시스템의 로그는 단순한 디버깅 기록이 아닙니다. 누가 어떤 요청을 했고, 모델이 어떤 도구를 선택했으며, 어떤 데이터가 반환됐고, 최종 변경을 누가 승인했는지 재구성할 수 있어야 합니다.

최소한 다음 항목은 남겨야 합니다.

- 요청자와 실행 시각
- 사용한 모델·도구·권한 범위
- 외부 데이터의 출처와 버전
- 제안, 승인, 실제 실행의 구분
- 실패·취소·롤백 기록

<InfoBox title="Design Principle">
  실행 가능한 AI에는 항상 **미리보기 → 승인 → 실행 → 검증 → 되돌리기**의 경로가 보여야 합니다.
  사용자에게 결과만 보여주면 자동화가 아니라 불투명한 위임이 됩니다.
</InfoBox>

## 4. AI 투자의 Phase 3: 가치 검증

에이전트 기술의 가능성이 커질수록 투자와 성과를 구분하는 기준도 더 엄격해져야 합니다. 인프라 규모나 데모의 인상만으로 실제 가치를 판단하기 어렵기 때문입니다.

| 단계               | 핵심 질문                        | 관찰할 지표                                       |
| :----------------- | :------------------------------- | :------------------------------------------------ |
| Phase 1 · 성능     | 모델이 무엇을 할 수 있는가?      | 정확도, 추론, 도구 사용 성공률                    |
| Phase 2 · 도입     | 조직에 연결할 수 있는가?         | 통합 비용, 보안, 사용자 채택                      |
| **Phase 3 · 가치** | **지속 가능한 결과를 만드는가?** | **시간 절감, 오류·사고율, 단위 경제성, 재사용률** |

재무적 관계를 분석할 때도 같은 원칙이 필요합니다. 투자금, 인프라 구매, 매출이 같은 생태계 안에서 순환하는 것처럼 보인다면, 단일 기사나 주가 변동보다 **공시, 계약, 현금흐름, 최종 고객 수요**를 함께 확인해야 합니다. 근거가 확인되지 않은 거래 구조를 사실처럼 단정하는 것은 거버넌스 분석 자체의 신뢰를 해칩니다.

## 결론: 성능이 아니라 운영 체계까지 평가하기

AI 에이전트의 경쟁력은 단순히 더 긴 작업을 수행하는 능력이 아닙니다. 실제 조직에서는 다음 세 질문에 답할 수 있어야 합니다.

1. 어떤 권한을 왜 부여했는가?
2. 결과와 책임을 사후에 재구성할 수 있는가?
3. 비용과 위험을 포함해 실제 가치가 검증되는가?

모델 이름과 벤치마크는 빠르게 바뀝니다. 반면 최소 권한, 사람의 승인, 감사 로그, 롤백, ROI 검증은 어떤 모델을 쓰더라도 남는 운영 원칙입니다.
